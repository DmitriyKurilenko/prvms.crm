# Архитектурные решения

## DEC-055: Планы продаж и аналитика воронки (2026-06-22, P2 Фаза 10)
**Контекст:** `Deal.loss_reason` собирался, но в аналитике не раскрывался; дашборд KPI существовал без целей менеджеров, конверсии по этапам и прогноза ([apps/crm/dashboard_api.py](../apps/crm/dashboard_api.py), [apps/crm/stats_api.py](../apps/crm/stats_api.py)). Фаза 10 добавляет планы продаж и четыре отчёта поверх уже собираемых данных, без внешних зависимостей.
**Решение:**
- **`closed_at` теперь заполняется.** Обнаружено, что `move_deal` ([apps/crm/deals_api.py:291](../apps/crm/deals_api.py#L291)) не выставлял `closed_at`, поэтому периодная аналитика по дате закрытия была невозможна. В `move_deal` добавлено: при входе сделки в стадию типа `won`/`lost` проставляется `closed_at = now()`, при возврате в `open` — обнуляется. Это инвариант, на котором стоят план/факт и прогноз; покрыт тестом.
- **Модель `SalesTarget`** (tenant, миграция `crm/0012`): план по менеджеру на месяц (`period_month`, `responsible`) с двумя метриками одновременно — `target_amount` (сумма выручки) и `target_count` (число выигранных сделок), любая может быть пустой. Уникальность по (`period_month`, `responsible`) — один план на менеджера в месяц; запись делается через `update_or_create` (upsert).
- **Отчёты — новый роутер `apps/crm/analytics_api.py`** (право `deals.view` + scope-фильтр `filter_crm_queryset_by_scope`, как в `stats_api`): `analytics/funnel/` (распределение по стадиям + доля + сводка won/lost/open + win-rate), `analytics/loss-reasons/` (группировка проигранных по `loss_reason`), `analytics/forecast/` (открытый пайплайн + сумма со `expected_close_date` в периоде), `analytics/target-progress/` (план против факта по менеджерам за месяц). CRUD планов `targets/` ограничен owner/admin.
- **Конверсия — на текущем распределении, не на истории.** `move_deal` логирует `stage_change` с названиями стадий, но без `stage_id`, поэтому точную поэтапную воронку из истории не восстановить надёжно. Воронка v1 строится на текущем распределении сделок по стадиям плюс win-rate (won/(won+lost)) — это честно и без парсинга свободного текста; ограничение зафиксировано в KNOWN_ISSUES #34.
- **Фронтенд:** `StatsView` расширен секциями «Воронка и конверсия» (KPI win-rate), «Прогноз закрытия» и «Причины потерь»; новый экран `SalesTargetsView` (план по менеджеру на месяц + прогресс-бары факт/план), маршруты и пункты меню «Аналитика» (всем) и «Планы продаж» (owner/admin — меню не фильтрует по роли, поэтому пункт гейтуется условием `isAdminOwner`).
**Альтернативы:** (1) считать факт по `updated_at` вместо `closed_at` — отклонено как неточное (любое касание сделки сдвигает `updated_at`); правильнее ввести honest `closed_at`. (2) Реконструировать поэтапную воронку из `stage_change`-активностей — отклонено как ненадёжное (нет `stage_id`). (3) План только по сумме или с разбивкой по воронкам — отклонено по согласованию: v1 — по менеджеру на месяц, метрики сумма и количество. (4) Гейтовать отчёты фичей `analytics` — отклонено ради единообразия с существующей CRM-аналитикой, гейтованной `deals.view`.
**Последствия и остаток (KNOWN_ISSUES #34):** воронка отражает текущее состояние, а не исторический проход по этапам; `closed_at` начинает заполняться только для будущих переходов (исторические закрытые сделки остаются с пустым `closed_at`, пока их не переместят); план — помесячный по менеджеру (без разбивки по воронкам и без командных целей сверх суммы менеджеров).

## DEC-054: Календарь, напоминания и повторяющиеся задачи (2026-06-22, P2 Фаза 9)
**Контекст:** Задача — это `Activity` с `due_date`/`status` ([apps/crm/models.py:164](../apps/crm/models.py#L164)); календарного представления, заблаговременных напоминаний и повторяемости не было (существовало только постфактумное событие `task_overdue`). Фаза 9 переиспользует планировщик Celery-beat из Фазы 5 и контур уведомлений.
**Решение:**
- **Модель.** В `Activity` добавлены `recurrence_rule` (RRULE по RFC 5545 без префикса, пусто = не повторяется; окончание серии несёт сама RRULE через UNTIL/COUNT, отдельного поля нет), `remind_at` (когда слать напоминание) и `reminder_sent_at` (идемпотентность). Миграция `crm/0011` + индекс `(status, remind_at)` для эффективного скана beat-задачи.
- **Повторяемость — внешний контракт `python-dateutil` сверен до кода** (версия 2.9.0): `rrulestr("RRULE:…", dtstart=due)` парсит правило с явным началом серии, `.after(dt, inc=False)` возвращает следующее вхождение или `None` при исчерпании. Логика изолирована в `apps/crm/services/recurrence.py::next_occurrence`; некорректная RRULE-строка не роняет закрытие задачи. Спавн следующего экземпляра происходит при переводе задачи в `done` внутри `patch_activity` ([apps/crm/activities_api.py](../apps/crm/activities_api.py)) — порождается ровно один экземпляр со сдвинутыми `due_date`/`remind_at`; зависимость `python-dateutil==2.9.0.post0` закреплена явно.
- **Напоминания — адресная доставка ответственному.** Существующий `notify()` рассылает по ролям ([apps/notifications/services.py:35](../apps/notifications/services.py#L35)), что не подходит для напоминания о конкретной задаче. Добавлен помощник `notify_user(tenant, user, event, context)`, целящийся в конкретного пользователя (ответственного), минуя ролевую рассылку, но соблюдая включённые каналы события. Beat-задача `send_task_reminders` (каждые 5 минут, [config/settings.py](../config/settings.py)) сканирует задачи `status='planned'`, `remind_at <= now`, `reminder_sent_at IS NULL`, шлёт напоминание и проставляет `reminder_sent_at` (идемпотентность по образцу `AutomationRunLog` из Фазы 5).
- **Новое событие `task_reminder`** добавлено в enum `NotificationEvent`. **Инвариант сидирования:** `seed_default_preferences()` срабатывает один раз на схему и не дозаполняет новые события — поэтому тенантам, засеянным до релиза, пара настроек добавляется идемпотентной data-миграцией `notifications/0004` (in_app включено, email выключено). Этот дефект был пойман живым зондом через реальный worker (напоминание ставило `reminder_sent_at`, но in-app `Notification` не создавалось из-за отсутствия настройки) и закрыт миграцией; alter-choices зафиксирован миграцией `notifications/0003`.
- **Календарь (фронтенд) — FullCalendar 6.1.21, а не vue-cal.** Изначально выбранная vue-cal оказалась EOL (v4.10.0 — финальный релиз, более не поддерживается) и без TypeScript-типов, её API только на внешнем docs-сайте — кодить против неё означало бы угадывать чужое поведение, что запрещено гейтом. В рамках решения «библиотека» взят официальный, типизированный (`dist/index.d.ts`) и поддерживаемый `@fullcalendar/vue3` + core/daygrid/timegrid/interaction (`peerDep vue ^3.0.11`, контракт сверён по README и `index.d.ts` установленного пакета). Новый `CalendarView.vue`: режимы месяц/неделя/день, диалог задачи с пресетами повторения (Ежедневно/Еженедельно/Ежемесячно → RRULE + сырое RRULE-поле), смещением напоминания (15 мин/1 час/1 день → `remind_at = due_date − offset`) и отметкой «выполнено». Источник данных — endpoint `GET /crm/activities/calendar/?date_from=&date_to=` (задачи ответственного в диапазоне). Задача из календаря назначается на текущего пользователя, иначе не попала бы в его выборку (дефект пойман и закрыт, подтверждён e2e).
**Альтернативы:** (1) vue-cal — отклонена как EOL/нетипизированная; (2) самописная сетка календаря — отклонена выбором пользователя в пользу библиотеки; (3) простой enum повторяемости вместо RRULE — отклонён выбором пользователя в пользу полного RFC 5545; (4) хранить дату окончания серии отдельным полем — отклонено, окончание несёт сама RRULE.
**Последствия и остаток (KNOWN_ISSUES #33):** календарь показывает задачи только текущего пользователя (командный вид и drag-and-drop переноса — следующий шаг); UI повторяемости даёт пресеты + сырое RRULE, но без визуального конструктора сложных правил; bundle вырос за счёт FullCalendar (gzip ~463 КБ, предупреждение >500 КБ было и раньше); реальная доставка письма-напоминания подтверждается только на боевом SMTP.

## DEC-053: Импорт/экспорт и дедупликация/слияние контактов и компаний (2026-06-22, P1 Фаза 6)
**Контекст:** В CRM не было загрузки клиентской базы из файла и выгрузки её обратно; экспорт существовал только для аудита (`apps/audit/api.py`). Дублирование контактов/компаний накапливалось без инструмента очистки. Фаза 6 закрывает импорт (CSV/XLSX), экспорт (CSV) и поиск/слияние дублей для двух базовых сущностей — контактов и компаний (сделки осознанно отложены: у них обязательные FK `pipeline`/`stage` и более сложный ключ дедупа).
**Решение:**
- **Парсер и экспорт — единый сервис `apps/crm/services/import_export.py`,** изолирующий stdlib `csv` и внешний `openpyxl`. Контракт `openpyxl==3.1.5` сверен с установленным пакетом до кода: `load_workbook(file, read_only=True, data_only=True)` принимает file-like, `ws.iter_rows(values_only=True)` отдаёт кортежи значений, в read-only режиме воркбук закрывается (`wb.close()`). Экспорт повторяет паттерн аудита (StringIO → `csv.writer` → `text/csv; charset=utf-8`) и добавляет BOM `﻿`, чтобы Excel корректно открывал кириллицу — это сознательное отличие от аудит-эндпоинта.
- **Импорт — фоновая Celery-задача `import_records`** (`apps/crm/tasks.py`), батчами по 50 строк, с построчной устойчивостью (`except` помечен `# noqa: BLE001`, ошибка строки пишется в `ImportJob.errors` с номером строки файла). Дедуп: контакты — по телефону, затем email; компании — по ИНН, затем названию. Колонки, отображённые на поле не из набора `allowed_fields`, складываются в `custom_fields`. Прогресс/результат поллит фронтенд через `GET /crm/import/jobs/{id}/`. **Инвариант:** одна новая модель `ImportJob` (миграция `crm/0010`), привязанная к тенант-схеме; задача принимает `schema_name` и работает через `schema_context`.
- **Слияние — сервис `apps/crm/services/merge.py`,** транзакционный (`@transaction.atomic`). Слияние контактов переносит `Deal.contact` и `Activity.contact` на основную запись; слияние компаний — `Contact.company` и `Deal.company`. Пустые поля основной записи дозаполняются из дублей, дубли удаляются. `primary_id` исключается из списка дублей. Поиск дублей группирует по тем же ключам (телефон/email, ИНН/название). **Слияние необратимо** — операция пишется в аудит через `log_event(action='update', changes={'Слияние дублей': ...})`.
- **Права переиспользуют существующие RBAC-сущности** `contacts`/`companies` (новая сущность и миграция `users` не вводятся): экспорт → `view` (плюс scope-фильтр `filter_crm_queryset_by_scope`, чтобы менеджер не выгрузил чужие записи), импорт → `create`, слияние → `delete` (плюс `ensure_crm_object_scope` на каждую участвующую запись).
- **Роутер `apps/crm/import_api.py`** подключён импортом в shim `apps/crm/api.py` (паттерн декораторов `crm_router`). Приём файла — `UploadedFile = File(...)` + `Form(...)` (паттерн `apps/tenants/api.py::upload_tenant_logo`); лимит строк — `_MAX_IMPORT_ROWS = 10000`.
- **Фронтенд — единый экран `DataToolsView.vue`** (`/app/data-tools`, пункт меню «Импорт/экспорт») с переключателем сущности и тремя вкладками: мастер импорта (загрузка → авто-сопоставление колонок → запуск → поллинг статуса с прогресс-баром и таблицей ошибок), экспорт (скачивание blob по паттерну `AuditView`), дубли (поиск групп + выбор основной записи + слияние).
**Альтернативы:** (1) хранить распарсенные строки на фронте между preview и run — отклонено, файл перезагружается в `run`, чтобы не держать большой массив в браузере; (2) отдельная RBAC-сущность `data_tools` — отклонено как лишняя миграция, естественнее переиспользовать права самих сущностей; (3) гонять импорт синхронно в запросе — отклонено, большие файлы должны идти фоном с поллингом.
**Последствия и остаток (KNOWN_ISSUES #32):** сделки не охвачены в v1; на импорте нет капчи/антиспама (эндпоинты под JWT и RBAC, не публичные); лимит 10 000 строк на файл; слияние необратимо (фронт подтверждает выбором основной записи, бэкенд пишет аудит); инлайн-парсинг XLSX-формул не выполняется (`data_only=True` берёт кэш-значения).

## DEC-052: Конструктор автоматизаций + SLA-триггеры (2026-06-22, P1 Фаза 5)
**Контекст:** Автоматизация ограничивалась полем `Stage.auto_action` (одно действие на стадию) без триггеров по времени/бездействию. Нужны настраиваемые правила «если → то», в т.ч. «нет активности N дней → задача».
**Решение:**
- Модели `AutomationRule` (`trigger`, `conditions` JSON, `action` JSON, `is_active`, `priority`) и `AutomationRunLog` (`unique(rule, deal)` — идемпотентность time-based правил), миграция `crm/0009`.
- **Переиспользуемый исполнитель:** из `process_stage_change` вынесена `execute_action(action, deal)` (`apps/crm/services/auto_actions.py`) с типами `create_task`/`send_notification`/`create_document`/`change_stage`/`assign`; `process_stage_change` стал обёрткой над ней (поведение сохранено — регресс-тест `test_auto_actions` зелёный).
- **Событийные триггеры:** `evaluate_event_rules(event, deal)` вызывается из `deals_api.create_deal` (`new_deal`) и `move_deal` (`stage_changed`). Рекурсии нет: события возбуждаются только из API-эндпоинтов, не из `execute_action`.
- **Time-based:** beat-задача `apps/crm/tasks.py::evaluate_time_rules` (`no_activity`, каждые 15 мин в `CELERY_BEAT_SCHEDULE`) обходит тенантов, находит open-сделки без активности дольше N дней, исключает уже сработавшие (`AutomationRunLog`), исполняет действие и логирует прогон.
- **API/Frontend:** CRUD `apps/crm/automation_api.py` (гейт `require_roles(['owner','admin'])`, валидация триггеров/действий); страница `AutomationView.vue` (конструктор «если → то»).
**Инварианты:**
- Защита от зацикливания: `change_stage`/`assign` в действии не возбуждают повторную событийную оценку; time-based правило срабатывает по сделке один раз (`AutomationRunLog`).
**Альтернативы:**
- Полноценный визуальный граф-билдер — отклонено для первой версии: достаточно линейного «триггер + условия + действие».
**Граница достоверности:** backend-тесты (`test_automation`: событие→задача, условие по воронке, идемпотентность time-rule; регрессия `test_auto_actions`) + e2e `automation.spec.ts` (`6 passed`): создание правила в ЛК и появление в списке. Триггер `sla_breach` и UI действий `change_stage/assign/create_document` в первой версии не выведены (бэкенд-исполнитель готов) — см. KNOWN_ISSUES.

## DEC-051: Теги и сегменты для контактов и сделок (2026-06-21, P1 Фаза 7)
**Контекст:** Не было способа помечать и группировать контакты/сделки для фильтрации и будущих рассылок (P2). Теги — фундамент сегментации.
**Решение:**
- Модель `Tag` (tenant-схема, `apps/crm/models.py`): `name` (уникальный), `color`, M2M к `Contact` и `Deal` с `related_name='tags'`. Модель `Segment` — именованный сохранённый фильтр (`entity` + `filters` JSON) как основа для аудиторий рассылок.
- CRUD и назначение — ninja `apps/crm/tags_api.py`: `tags/` CRUD, `segments/` CRUD, `contacts/{id}/tags/` и `deals/{id}/tags/` (PATCH `tag_ids`). Гейт существующей сущностью прав `contacts`/`deals` — отдельная RBAC-сущность не вводится (теги — метаданные контактов/сделок).
- Фильтрация списков: `list_contacts`/`list_deals` приняли параметр `tag_id` (`.filter(tags__id=...)`).
- Frontend: страница «Теги» (`TagsView.vue`, создание/список/удаление с цветом), пункт меню, типы/функции в `api/crm.ts` (включая `setDealTags`/`setContactTags`).
**Инварианты:**
- Имя тега уникально в пределах тенанта (`UniqueConstraint`).
- Сегмент хранит только конфигурацию фильтра, без персональных данных.
**Альтернативы:**
- Отдельная сущность прав `tags` — отклонено: теги логически принадлежат контактам/сделкам, дублировать RBAC-матрицу избыточно.
**Граница достоверности:** backend-тесты (`test_tags`: назначение и фильтрация по тегу для контактов и сделок, уникальность имени, хранение фильтра сегмента) + браузерный e2e `tags.spec.ts` (`5 passed`): создание тега в ЛК и появление в списке. UI назначения тегов в карточке контакта/сделки и UI сегментов в первой версии не сделаны (бэкенд готов) — см. KNOWN_ISSUES.

## DEC-050: Веб-формы захвата лидов + встраиваемый виджет (2026-06-21, P1 Фаза 4)
**Контекст:** На лендинге платформы форма обратной связи была, но тенант не мог создать собственную форму для своего сайта; лиды с сайта вводились вручную. Внешние CRM (amoCRM, Битрикс24) такое умеют.
**Решение:**
- Модель `WebForm` (tenant-схема, `apps/crm/models.py`): `public_token` (UUID), `fields_schema` (JSON-описание полей), pipeline/stage, `source`, `auto_distribute`, `success_message`, `allowed_origins`, `submissions_count`. Резолв тенанта по публичному токену — shared-lookup `WebFormLookup` (public-схема, `apps/tenants/models.py`), создаётся/удаляется в одной транзакции с формой.
- Публичный приём (`apps/crm/public_views.py`, подключён в `config/urls.py`): `POST /api/public/webform/<uuid:token>/` и `GET …/schema/`. ## DEC-045: Доставка формы обратной связи лендинга по SMTP (2026-06-21)
**Контекст:** На лендинге есть форма «Написать нам», которая POST-ит на `/api/public/pricing/telephony-request/`. Обработчик сохранял заявку в БД и ставил письмо в очередь, но письмо никуда не доходило: `EMAIL_BACKEND` был `console` (печать в лог воркера), в `settings.py` не были проброшены SMTP-параметры транспорта, а выделенной env-переменной адреса-получателя не существовало. Хостинг (Beget) задаёт креды в nodemailer-стиле (`SMTP_HOST/SMTP_PORT/SMTP_SECURE/SMTP_USER/SMTP_PASS`, `CONTACT_TO/CONTACT_FROM`), а не в Django-именах `EMAIL_*`.

**Решение:**
- В `config/settings.py` env-переменные хостинга маппятся на Django-контракт почты: `SMTP_HOST→EMAIL_HOST`, `SMTP_PORT→EMAIL_PORT`, `SMTP_USER→EMAIL_HOST_USER`, `SMTP_PASS→EMAIL_HOST_PASSWORD`, `CONTACT_FROM→DEFAULT_FROM_EMAIL`, `CONTACT_TO→SUPPORT_EMAIL` (новая настройка-получатель).
- Семантика nodemailer `secure` приведена к взаимоисключающим флагам Django: `SMTP_SECURE=true → EMAIL_USE_SSL=True` (неявный TLS, порт 465), `false → EMAIL_USE_TLS` (STARTTLS, 587). Деривация `EMAIL_USE_TLS = (not EMAIL_USE_SSL) and host not in ('', 'localhost')` гарантирует требуемую `smtp.py` взаимоисключаемость и не включает STARTTLS для `console`/localhost.
- Обработчик `apps/billing/public_views.py` различает источник заявки (`source == 'landing-contact'` против калькулятора телефонии): осмысленная тема, человекочитаемое тело с полями «Имя/Телефон/Email/Сообщение» (а не JSON-дамп), получатель `SUPPORT_EMAIL or DEFAULT_FROM_EMAIL`.
- Самодиагностика: `public_views` логирует постановку письма и предупреждает при незаданном получателе; `send_email_async` логирует `delivered=<N>` либо ловит узкие `(smtplib.SMTPException, OSError)`, пишет traceback и пробрасывает исключение.

**Последствия:**
- Форма обратной связи доставляет письмо на адрес `CONTACT_TO` через SMTP; данные заявки по-прежнему сохраняются в `TelephonyQuoteRequest` независимо от исхода доставки.
- Инвариант: транспорт почты конфигурируется только через env; смена провайдера не требует правок кода.
- Устранён молчаливый fallback на `console`-бэкенд: `settings.py` автоматически выбирает SMTP, если задан `SMTP_HOST`, но `EMAIL_BACKEND` не задан; `manage.py check` предупреждает, когда активен `console` при настроенном SMTP-хосте.
- В production-стеке `celery` подключён к внешней сети `traefik`, чтобы исходящие SMTP-соединения не блокировались `internal: true` сети `backend`.
- Задача `send_email_async` использует `autoretry_for=(smtplib.SMTPException, OSError)` с экспоненциальным бэкоффом для переживания кратковременных сетевых сбоев.
- Граница проверена живым зондом (Beget принял сообщение, `delivered=1`); фактический приход в ящик подтверждается получателем.

## DEC-042: Телефония перенесена на облако MTS Exolve, FreeSWITCH удалён (2026-06-15)
**Контекст:** Прежняя телефония работала на самостоятельном FreeSWITCH (dialplan/directory/ESL через `xml_curl`, sip.js-софтфон, отдельный docker-сервис с SIP/RTP/ESL-портами). Контур был помечен экспериментальным (KNOWN_ISSUES #2), требовал сопровождения медиа-сервера и не давал «телефонию из коробки» для множества тенантов. Задача: новый канал-телефония через MTS Exolve, всё автоматически, голос в браузере, входящий создаёт сделку с контролем дублей и переадресуется ответственному.
**Решение:**
- **FreeSWITCH удалён полностью**: сервис из `docker-compose.yml`, `docker-compose.telephony.yml`, каталог `freeswitch/`, env-блок `FREESWITCH_*`/`SIP_BASE_DOMAIN`, ESL-код, sip.js и `useSIPPhone.ts`, публичные XML-эндпоинты (`dialplan/directory/events/configuration`). Зависимость `greenswitch` убрана.
- **Облако Exolve** через официальные контракты: Numbering API (`GetFree → Lock → Buy → SetCallForwarding`), SIP API (`Create → GetAttributes → SetDisplayNumber`), Web Voice SDK (`@mts-exolve/web-voice-sdk`), Call Events webhook.
- **Один номер на тенант**, закупается через API. Пользователь только выбирает номер в разделе «Телефония» (мастер `ExolveNumberWizard`); бронь, покупка и настройка переадресации — автоматические. Резолв тенанта по номеру — shared-модель `tenants.ExolveNumberLookup` (паттерн `SigningTokenLookup`).
- **Внутренние номера менеджеров** = SIP-аккаунты Exolve (`telephony.ExolveSIPAccount`), создаются автоматически, CLI = номер тенанта. Браузер менеджера регистрируется через Web Voice SDK; глобальный софтфон — Pinia-store `stores/phone.ts` + `SoftPhone.vue` в `App.vue`.
- **Входящий звонок** обрабатывается синхронно в IPCR-хуке `getControlCallFollowMe` (`POST /telephony/exolve/ipcr/`): резолв тенанта по `sip_id` (вызванный номер), контакт по `numberA`, **контроль дублей** (при наличии сделки в стадии `stage_type='open'` новая не создаётся), определение ответственного и `followme_struct` (`redirect_type=2`: ответственный order 1 → остальные активные SIP order 2). Возвращается JSON-RPC.
- **Журнал и записи** — Call Events (`POST /telephony/exolve/events/`, события `b/o/s/h/d/e/crr`) → `CallRecord` (ключ `call_sid`), запись скачивается фоном через Bearer.
**Инварианты:**
- `EXOLVE_API_KEY` — один на платформу; webhook-и защищены `EXOLVE_WEBHOOK_SECRET` (`?key=…`).
- «Активная сделка» для дедупа = сделка в стадии с `stage_type='open'`.
- При ошибке решения по входящему IPCR возвращает пустой `followme_struct` — звонок не теряется (Exolve уходит на `reserve`).
- Домен SIP всегда `sip.exolve.ru`; исходящий CLI задаётся `SetDisplayNumber` = номер тенанта.
**Альтернативы:**
- Сохранить FreeSWITCH и подключить Exolve как SIP-транк — отклонено: требует сопровождения медиа-сервера и не даёт динамическую маршрутизацию на ответственного.
- Статическая переадресация номера на один SIP — отклонено: не покрывает требование «ответственному за контакт» (динамика на звонок).
- Автоподбор номера без участия пользователя — отклонено по требованию: выбор номера делает пользователь в UI, остальное автоматически.
**Граница достоверности:** реальный голосовой звонок и фактические ответы Exolve проверяются только на проде с боевым ключом; контур самодиагностируем (полный лог IPCR/событий/провижининга). См. KNOWN_ISSUES.

## DEC-040: Production-стек переведён с nginx на shared Traefik reverse proxy (2026-06-01)
**Контекст:** Проект `prvms.crm` ранее использовал собственный контейнер `nginx:alpine` для терминации TLS, обслуживания статики и reverse-proxy к `web` / `frontend-app`. Это занимало порты 80/443 на хосте, требовало ручного управления SSL-сертификатами и конфликтовало с другими проектами на том же сервере.
**Решение:** Внешний трафик теперь принимает единый shared Traefik (запущенный один раз на сервере). Контейнеры проекта объявляют себя через Docker labels (`traefik.http.routers.*`) и подключаются к внешней сети `traefik`. Проект не открывает порты 80/443 и не содержит собственного reverse proxy.
**Маршрутизация:**
- `prvms-api` (priority 100) — API, WebSocket, admin, webhooks, media, healthcheck, **root `/` (landing page)** → `web:8000`.
- `prvms-static` (priority 50) — `/static` (Django collected static) → `frontend-app:80`.
- `prvms-spa` (priority 1) — catch-all для Vue SPA → `frontend-app:80`.
**Альтернативы:**
- Оставить nginx + выделить проекту отдельный IP — отклонено, shared-хостинг без дополнительных IP.
- Использовать Docker Swarm ingress + overlay — отклонено, стек работает в compose-режиме.
**Инварианты:**
- Имена роутеров (`prvms-api`, `prvms-static`, `prvms-spa`) уникальны глобально на сервере (префикс проекта).
- Все сервисы, reuse образ `web`, имеют explicit `image: prvms-crm-web:latest` — никаких implicit pull с Docker Hub.
- `frontend-app` **не имеет healthcheck**. Traefik 2.x намеренно не регистрирует роутеры для unhealthy/starting контейнеров; busybox-wget probes создают больше outage, чем предотвращают. Без healthcheck Traefik регистрирует роутер немедленно; Docker restart policy восстанавливает nginx.
- `migrate` запускает `collectstatic` до старта `web`/`frontend-app`, гарантируя что `/static/` работает с первого запроса.
- `deploy.sh` проверяет наличие сети `traefik` и запущенного контейнера `traefik` перед деплоем.
- Проект запускается только через `./deploy.sh`; прямой `docker compose up -d` неприменим, так как образы требуют предварительного `build`.

## DEC-017: CRM — ответственный привязан к User, а не к ManagerProfile (2026-04-15)

## DEC-018: Простая электронная подпись (ПЭП) для договоров (2026-04-16)
**Контекст:** Подписанные договоры не имели криптографической защиты — статус «подписан» менялся только в БД без привязки к содержимому документа.
**Решение:** Реализована простая электронная подпись (ПЭП) по 63-ФЗ. При создании контракта вычисляется SHA-256 хеш PDF-файла (`Contract.pdf_hash`). При подписании создаётся HMAC-SHA256 подпись, включающая хеш документа, данные подписанта (телефон, IP, User-Agent), timestamp и ID сессии (`Contract.signature_data` JSONField). OTP генерируется через `secrets` (криптографически безопасно).
**Последствия:** Документ получает юридическую значимость в рамках ПЭП. При проверке можно верифицировать: а) целостность PDF (сравнение хешей), б) подлинность подписи (HMAC через SECRET_KEY). Для усиленной подписи (УНЭП/УКЭП) потребуется интеграция с CryptoPro или аналогом — это отложено на production-hardening.

## DEC-019: Real-time чаты через Django Channels, не polling (2026-04-15)

**Решение:** `ChatConsumer` (AsyncJsonWebsocketConsumer) на `ws/chat/?token=...&slug=...`. JWT-аутентификация + tenant slug через query params в `JWTQueryAuthMiddleware`. Celery-задачи (`route_incoming_message`, `route_outgoing_message`) после создания `MessageLog` делают `channel_layer.group_send` для push-доставки.
**Инвариант:** Группы WS включают tenant slug: `chat.{tenant_slug}.channel.{channel_id}`. Это обеспечивает изоляцию между тенантами — channel_id уникален только внутри tenant schema.
**Последствия:** Никакого polling. Сообщения доставляются мгновенно через WS. При отсутствии slug в query WS-соединение отклоняется (code 4400).

## DEC-034: HealthCheckBypassMiddleware + IPv4-литерал для liveness-probe (2026-05-11)
**Контекст:** Диагностика DEC-033 устранила симптом (Traefik не видел роутеры), но при следующем деплое сертификат снова не выдался. Debug-лог Traefik показал `Filtering unhealthy or starting container` для `crm_prvms-web` и `crm_prvms-frontend-app` — Traefik 2.x **намеренно** не регистрирует роутеры контейнеров со статусом `unhealthy`/`starting`. Причины unhealthy оказались две, обе системные:
1. `web` healthcheck бил `curl http://localhost:8000/healthz`, но Django возвращал 404 — `django_tenants.middleware.main.TenantMainMiddleware` стоит перед URL-резолвом и не находит домен `localhost`/`127.0.0.1` в shared schema `Domain`, поэтому отдаёт 404 ещё до того, как `config/urls.py` получает шанс отработать. `SHOW_PUBLIC_IF_NO_TENANT_FOUND=True` поведение не меняет в актуальной версии django-tenants в этой конфигурации.
2. `frontend-app` healthcheck использовал `wget -q --spider http://localhost/`. Busybox-`wget` в `nginx:alpine` резолвит `localhost` в `::1` первым и **не** делает fallback на IPv4, а nginx слушает только IPv4 (entrypoint `10-listen-on-ipv6-by-default.sh` пропускает добавление IPv6-listen из-за «differs from packaged» конфига).

**Решение:**
- `apps/core/middleware.py`: новый `HealthCheckBypassMiddleware`, отвечает `JsonResponse({'status':'ok'})` на `/healthz` и `/healthz/` **до** любых других middleware.
- `config/settings.py`: `HealthCheckBypassMiddleware` поставлен первым в `MIDDLEWARE`, до `TenantMainMiddleware`.
- `vps-deployment/crm_prvms/docker-compose.yml`: healthcheck `frontend-app` использует `http://127.0.0.1/` вместо `localhost` (IPv4-литерал, обход busybox-wget без IPv6-fallback).
- `vps-deployment/scripts/start-all.sh`: добавлен preflight `crm_prvms`: если `PUBLIC_HOSTNAME` отсутствует/пуст в `.env.prod`, проект не стартует — Traefik-лейблы шаблонизированы на этой переменной и без неё дают `Host(``)`.
- `.gitignore`: убран блок-исключение `/vps-deployment` (тёрло отслеживание полезных файлов); добавлены прицельные паттерны `vps-deployment/**/.env*`, `acme.json`, `logs/`, `media/`. Удалён случайный снапшот `vps-deployment/crm_prvms/.venv.current_on_server` с production-секретами.

**Последствия:**
- Liveness-probe больше не зависит от состояния тенантов/доменов — endpoint работает с любого `Host` header, до tenant resolution.
- Контейнер `frontend-app` становится healthy через 30 секунд после старта без костылей с curl/wget внутри образа.
- `start-all.sh` обрывает запуск CRM до того, как Traefik увидит контейнер с пустым `Host(``)` лейблом — fail-fast вместо silent-fail.
- Любая утечка `.env.prod` или серверного снапшота в репозиторий заблокирована на уровне `.gitignore`.
- DEC-033 (перезапуск Traefik) остаётся как defensive measure — устраняет редкие случаи пропуска docker-events. Не отменяется.

### Дополнение (2026-05-11, после прод-выкатки v1):
v1 не закрыла проблему до конца — на сервере `/` всё равно отдавал 404. Причины и доводка:
- **`/opt/crm_prvms/docker-compose.yml` — копия, не симлинк.** При initial setup кто-то скопировал файл вместо `ln -sf`. Git pull обновлял источник (`vps-deployment/crm_prvms/docker-compose.yml`), но рабочий compose оставался прежним → новый healthcheck не попадал в контейнер. Структурный фикс: `deploy.sh` и `start-all.sh` теперь идемпотентно пересоздают симлинки при каждом запуске, копии бэкапятся в `*.copy_replaced_*.bak`. Эта ловушка больше невозможна.

## DEC-035: Messenger incoming pipeline — explicit failure visibility + update-type filtering (2026-05-13)
**Контекст:** Сделки из Telegram/MAX не создавались при входящих сообщениях. Аудит показал несколько системных проблем: (1) `normalize_incoming_payload` для Telegram не обрабатывал `edited_message` — весь update попадал в `payload`, `chat_id` становился `'unknown'`, сообщения терялись или сливались в одну сессию; (2) MAX `bot_started` update создавал пустую сессию с `chat_id='unknown'`; (3) при отсутствии Pipeline или Stage `auto_create_lead` молча пропускал создание сделки без логирования и без записи ошибки в UI; (4) `except Exception` в `tasks.py` проглатывал реальные баги, оставляя ops без диагностики.

- `apps/channels/providers.py`: `normalize_incoming_payload` теперь возвращает `dict | None`. ## DEC-037: Адаптивный UI — single-source responsive layer + class-agnostic card-таблицы (2026-05-16)
**Контекст:** В мобильной версии боковое меню не скрывалось. Корневая причина — **баг CSS-специфичности**, а не JS-логики: десктопное правило `.layout-static .layout-sidebar { transform: translateX(0) }` (специфичность `0,2,0`) всегда перебивало мобильное `@media (max-width:991px) .layout-sidebar { transform: translateX(-100%) }` (специфичность `0,1,0`) — media query не добавляет специфичности, а контейнер всегда несёт класс `layout-static`. Сайдбар был постоянно виден на телефоне, перекрывал контент, а переключение `layout-mobile-active` визуально ничего не делало (оба состояния резолвились в `translateX(0)`). Дополнительный аудит показал, что адаптации не было системно: `.form-row-2` (1fr 1fr) не схлопывался (дублировался scoped в 8 файлах), `.section-header`/тулбары/tab-bar не переносились, 24 `PDataTable` переполнялись по ширине, диалоги с фикс-шириной 520–900px выходили за вьюпорт, layout `220px/280px 1fr` (Tasks/Assistant) не сворачивался.
**Решение (комплексное, single source of truth — без `!important`-заплаток по месту):**
- **Структурный фикс layout:** десктопные layout-mode-правила вынесены в `@media (min-width: 992px)`, мобильный off-canvas — в `@media (max-width: 991px)`. Диапазоны взаимоисключающие → коллизия специфичностей устранена структурно (десктопных селекторов ниже 992px просто нет). JS (`useLayout.toggleMenu/hideMobileMenu`, watcher `route.path` в `AppSidebar`) не менялся — он был корректен.
- **Глобальный адаптивный слой в `styles/main.css`:** `.form-grid/.form-row-2/.form-row-3` стали глобальными примитивами с responsive-сворачиванием (≤640px → 1 колонка); продублированные scoped-определения удалены из 6 файлов (scoped-правило с `[data-v-*]` имеет специфичность `0,2,0` и затеняло бы глобальный media-override). `.section-header` переносится на ≤640px. `.p-dialog`/`.p-drawer` получили `max-width: 95vw` глобально (одна точка вместо 12 per-dialog `maxWidth`). Topbar: org-select и кнопки сжимаются на ≤991px, wordmark скрывается на ≤480px.
- **Карточный режим таблиц (директива `v-responsive-table`):** `frontend/src/directives/responsiveTable.ts` помечает корень таблицы классом `.rt-cards` и копирует текст заголовка колонки в `td[data-label]`; на ≤767px глобальный CSS превращает каждую строку в карточку «Лейбл: значение», строку пустого сообщения (одна ячейка с `colspan`) помечает `.rt-empty-row` и оставляет одним блоком. Селекторы **PrimeVue-class-agnostic** (семантические `thead>tr>th` / `tbody>tr>td`), проверены против реального DOM PrimeVue 4.4 — минорный апгрейд PrimeVue не сломает раскладку. Ре-синк: Vue `updated` + `MutationObserver` (childList) на сорт/пагинацию/async-рендер; без петли (директива меняет только атрибуты, observer слушает childList). Применена ко всем 24 `PDataTable`.
- **Per-view добивка:** `tasks-layout`/`assistant-layout` сворачиваются в 1 колонку на ≤768px (на телефоне список диалогов ассистента ограничен по высоте, чтобы чат оставался доступен); `.tabs-bar`/`.tab-bar` (Contracts/Team/Pipelines) получили `flex-wrap`+`overflow-x`.
**Последствия:**
- Боковое меню корректно работает off-canvas на ≤991px (drawer + mask + закрытие по клику/навигации); десктопный static/overlay не затронут.
- Адаптивность повторяющихся примитивов поддерживается в одном месте — новые view, использующие `.form-row-2`/`.section-header`/`PDataTable`+`v-responsive-table`, адаптивны без дополнительного кода. **Инвариант:** компоненты не переопределяют `.form-grid/.form-row-2/.form-row-3` в scoped-стилях.
- Валидация: `manage.py check` — 0 issues; `npm run typecheck` EXIT=0; `npm run build` EXIT=0; **5/5 vitest**; в собранном бандле присутствуют `rt-cards`/`min-width:992px`/`max-width:991px`/`data-label` (CSS прошёл через Vite), директива забандлена; dev SPA `/app` → 200.
- Браузер-QA на реальном устройстве не выполнялся в этой среде (нет браузера); фикс специфичности детерминирован (доказуем по правилам каскада), DOM-предположения директивы сверены с исходниками PrimeVue 4.4 — см. KNOWN_ISSUES.

## DEC-038: Реструктуризация навигации CRM — сделка как страница, чат внутри, меню перегруппировано (2026-05-17)
**Контекст:** Пользовательский тест показал, что модальное окно сделки не позволяет дать ссылку на конкретную сделку, чат с клиентом открывался в отдельном разделе «Чаты» вместо контекста сделки, а боковое меню содержало слишком много пунктов (Мессенджеры, Распределение, Уведомления, Интеграции) которые используются реже чем Чаты.
**Решение:**
- **Сделка — полноценная страница:** `/app/deals/:id` (динамический роут). `DealDetailView.vue` с тремя табами: Инфо, Активность, Чат. Все данные загружаются по id из URL; backward-совместимость — старые вызовы `openDeal(id)` в `DealsView` заменены на `router.push`. Удалён `DealDetailDialog.vue` (202 LOC).

Переключение канала — через `PSelect` по `CrmDealChatSessionRef`. Дедупликация сообщений, optimistic update, auto-scroll — паттерн 1:1 с `ChatsView`.
- **Контакт — связанные сделки:** `ContactDrawer` получил таб «Сделки» с загрузкой через `contactDeals(id)` (фильтр `listDeals({ contact_id })`). Переход на `/app/deals/:id`.
- **Меню — перегруппировка по частоте использования:**
  - Группа 1 (CRM ядро): Дашборд, Контакты, Компании, Сделки, Задачи.
  - Группа 2 (Коммуникации): Чаты (вместо Мессенджеров), Договоры, Телефония.
